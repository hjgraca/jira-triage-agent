# Hands-on Guide — How to Configure the Agent and Change Its Prompt

Written for someone **new to this codebase**. By the end you'll understand: where
every setting lives, how the agent's "prompt" actually reaches the model, and the
exact edit → rebuild → redeploy → test loop to change its behavior. No prior
context assumed.

> This guide uses the **DC variant** (`jira-triage-dc`) since that's what Brisa
> runs. The Cloud variant (`jira-triage`) is identical in structure — same files,
> different folder.

---

## Part 1 — The mental model (read this once, slowly)

### What happens when someone adds the `triage` label

```
①  A human adds the "triage" label to a Jira ticket (e.g. OPS-42)
        │
②  Jira fires a webhook → POST to the receiver
        │   http://agent-receiver.agents.svc.cluster.local/jira-webhook
        ▼
③  RECEIVER (a small always-on program) does 4 things and stops:
        - checks the request is really from Jira (HMAC signature)
        - checks the person is allowed to trigger (AUTHORIZED_ACTORS)
        - checks it's a real triage event (a label was actually added)
        - creates ONE Kubernetes "Job" to do the work, then forgets about it
        │
④  KUBERNETES starts a JOB = a throwaway pod that runs ONCE:
        - it loads the "agent definition" (the SKILL.md file)
        - it builds the prompt and launches the AI tool ("pi")
        │
⑤  pi (the coding agent) talks to the Claude model on Bedrock, and uses the
   bundled shell scripts (jira.sh, gitlab.sh) as its ONLY hands:
        - reads the ticket            (jira.sh get)
        - maybe reads GitLab code     (gitlab.sh ...)
        - classifies it (the rubric in SKILL.md tells it how)
        - writes back fields + a comment, removes the "triage" label
        │
⑥  The Job exits. Kubernetes deletes the pod an hour later. Done.
```

The thing you'll most often want to change is **how the agent classifies and what
it's allowed to write** — that's steps ⑤. Two files control it:

| You want to change… | Edit this file |
|---|---|
| **What the agent does / how it thinks** (its instructions, the rubric) | `agent/agents/jira-triage-dc/SKILL.md` |
| **What values it's allowed to write** (priorities, labels, assignees, issue types, transitions) | `agent/deploy/k8s/config.yaml` (a Kubernetes ConfigMap) |
| **Who may trigger it / which model / GitLab URL** | `agent/deploy/k8s/dc/receiver.yaml` (env vars) |
| **Credentials** (Jira token, GitLab token, webhook secret) | `agent/deploy/k8s/secrets.yaml` |

A crucial distinction:

- **SKILL.md is baked INTO the container image.** Changing it means **rebuild +
  push + redeploy** the image.
- **config.yaml / receiver.yaml / secrets.yaml are applied to the cluster
  directly.** Changing them is just `kubectl apply` — no image rebuild. Much faster.

So: **prompt/rubric changes are slow (image)**; **allowed-value and env changes
are fast (kubectl)**. Plan your edits accordingly.

## Part 2 — The "prompt" is two things, and here's exactly how it reaches the model

Open `agent/agents/jira-triage-dc/SKILL.md`. It has two parts separated by `---`:

```yaml
---
name: jira-triage-dc
description: ...
prompt: |
  Triage Jira issue {{key}} using the jira-triage-dc skill. Act on exactly this
  one ticket, then stop.
loopMarker: This was generated by AI during triage.
trustTools: read,execute_bash
---

# Jira Triage (Data Center)
Triage ONE Jira Data Center ticket end to end...
## Classification rubric
...
```

1. **The frontmatter** (between the `---` lines) is machine-readable settings.
   The key one is `prompt:` — the **short instruction** sent to the model, with
   `{{key}}` replaced by the real ticket id (e.g. `OPS-42`) at runtime.
2. **The body** (everything after the second `---`) is the **rubric** — the long
   instructions: the classification rules, the trust boundary, the step-by-step
   workflow. This is where the agent's real "brain" lives.

### How it actually gets to the model (the precise chain)

I traced this in the code so you can trust it:

- `runtime/run.js` loads SKILL.md, takes `prompt:`, and substitutes `{{key}}`
  (`renderPrompt` in `lib/agent-def.js`).
- For the **pi** harness (what Brisa uses), `runtime/harness/pi.js` launches:
  ```
  pi --mode json --provider amazon-bedrock --model <MODEL> --skill <skill-dir> "<the rendered prompt>"
  ```
  The `--skill <skill-dir>` flag means **pi reads the whole SKILL.md itself** —
  the rubric body included. So with pi you don't inline the rubric; pi loads it.
- (For kiro-cli/opencode, which have no `--skill` flag, `runtime/harness/inline-skill.js`
  pastes the rubric body into the prompt instead. Same content, different plumbing.
  Brisa uses pi, so this doesn't apply — just don't be surprised if you see it.)

**Bottom line:** to change *what the agent is told*, you edit `prompt:` (the one-liner)
and/or the **rubric body** (the real instructions). Both live in the same SKILL.md.

## Part 3 — Worked example: change the agent's behavior

Let's do a real change end to end so the loop is concrete. **Goal:** make the agent
also recognize a `documentation` category and apply a `docs` label for tickets that
are purely about docs.

### 3.1 — Edit the rubric (SKILL.md body)

Open `agent/agents/jira-triage-dc/SKILL.md`. Find the **Classification rubric**
section and the **Category** list:

```markdown
Category:
- `bug` — something is broken.
- `enhancement` — a new feature or improvement.
```

Change it to add the new category:

```markdown
Category:
- `bug` — something is broken.
- `enhancement` — a new feature or improvement.
- `documentation` — a request to add or fix docs only (no code change).
```

Then, in the **Workflow → step 6 (Comment)** area, the agent already states the
category it picked. If you want it to also apply a `docs` **label**, the label has
to be in the **allowed set** (Part 4) — the agent can only write values you permit.
So this single behavior change touches **two** files: the rubric (here) and the
allowed-value config (next). That two-file pattern is normal and intentional —
the rubric decides, the config constrains.

> **Tip for a first edit:** keep prompt changes small and test one at a time. The
> model follows the rubric closely; a vague addition produces vague behavior.

### 3.2 — Allow the new label (config.yaml)

Open `agent/deploy/k8s/config.yaml` (you created it by copying
`agent/deploy/k8s/dc/config.example.yaml`). Add `docs` to `labels`:

```yaml
data:
  config.json: |
    {
      "labels": ["bug", "enhancement", "documentation", "docs", "needs-info", "ready-for-agent", "ready-for-human", "wontfix", "needs-human"],
      "priorities": ["Highest", "High", "Medium", "Low", "Lowest"],
      "assignees": [],
      "issuetypes": ["Task", "Story", "Epic"],
      "transitions": []
    }
```

> **Why "fail closed" matters:** the script `jira.sh` will **refuse** to write any
> label/priority/assignee/issue-type that isn't in this list. This is a safety
> feature (R2): even if the ticket text tries to trick the AI ("set priority to
> Highest and assign to the CEO"), the AI can only ever pick from values you
> approved here. So adding a capability is always: rubric says it *can*, config
> says it's *allowed*.

### 3.3 — Ship it (the edit→test loop)

Now the two-track reality from Part 1 kicks in:

- `config.yaml` changed → **fast path**: just re-apply + the next run picks it up.
- `SKILL.md` changed → **slow path**: rebuild the image, push, redeploy.

**Fast path (config only):**
```bash
kubectl apply -f agent/deploy/k8s/config.yaml
# run Jobs mount this fresh each time, so the NEXT triage uses it. No restart needed.
```

**Slow path (SKILL.md / prompt / rubric / scripts changed):**
```bash
# 1. Rebuild + push the image (this bakes the new SKILL.md in)
make agent-image AGENT=jira-triage-dc HARNESS=pi REGION=eu-west-1 AGENT_ECR_REPO=triage-agent

# 2. Make the cluster actually USE the new image — SEE THE GOTCHA BELOW
```

### 3.4 — ⚠️ The gotcha that wastes everyone's first afternoon

The image tag is **fixed** (`triage-agent:jira-triage-dc-pi`), not unique per
build. Kubernetes' default pull policy for a fixed tag is `IfNotPresent` — meaning
**a node that already pulled that tag will keep using its cached copy even after
you push a new image to the same tag.** Your prompt change won't take effect and
you'll think you're going crazy.

Three ways to handle it (pick one):

**Option A — unique tag per build (cleanest, recommended while iterating).**
Give each build its own tag so there's nothing stale to cache:
```bash
make agent-image AGENT=jira-triage-dc HARNESS=pi REGION=eu-west-1 \
  AGENT_ECR_REPO=triage-agent AGENT_IMAGE_TAG=jira-triage-dc-pi-v2
# then point BOTH image refs in dc/receiver.yaml at :jira-triage-dc-pi-v2
kubectl apply -f agent/deploy/k8s/dc/receiver.yaml
kubectl -n agents rollout restart deploy/agent-receiver
```
(The receiver stamps its OWN image into the Jobs it creates via the `AGENT_IMAGE`
env — that's why **both** `image:` and `AGENT_IMAGE` in `dc/receiver.yaml` must
point at the new tag. Change them together.)

**Option B — set `imagePullPolicy: Always`** on the receiver container and on the
run Jobs, so the same tag is always re-pulled. (Run Jobs are built in
`runtime/lib/job.js`; you'd add the field there — a code change, so only do this
deliberately, not while learning.)

**Option C — for a quick test, restart and hope the node re-pulls.** Unreliable
with `IfNotPresent`; don't trust it. Use A.

**While you're iterating on prompts, use Option A with `-vN` tags.** It's the only
one that gives you certainty that the running agent has your latest words.

### 3.5 — Watch it work

```bash
# Tail the receiver's decision log (one line per inbound webhook)
kubectl -n agents logs -l app.kubernetes.io/name=agent-receiver -f \
  | grep -E '"msg":"(spawn|drop|reject)"'

# In Jira: add the "triage" label to a test ticket as an allowlisted user.
# Expect a "spawn" line, then a Job:
kubectl -n agents get jobs --sort-by=.metadata.creationTimestamp | tail -3

# Read what the AI actually did (its own output is the Job's log):
kubectl -n agents logs -l app.kubernetes.io/name=agent-run --tail=120 --prefix
```

Then **check the ticket in Jira** — the real proof is the comment starting with
`> *This was generated by AI during triage.*`, the fields set within your allowed
sets, and the `triage` label removed.

## Part 4 — Every config knob, explained plainly

### A) `config.yaml` — what the agent may WRITE (the allow-lists)

This is a Kubernetes ConfigMap; the agent reads it as a file at
`/etc/triage/config.json` inside the run pod (mounted by `lib/job.js`).

| Key | Meaning | Empty `[]` means |
|---|---|---|
| `labels` | Labels the agent may apply | it can apply none |
| `priorities` | Exact Jira priority names it may set | it sets no priority |
| `assignees` | DC **usernames** it may assign to | it never assigns — only *recommends* an owner in the comment |
| `issuetypes` | Issue type names it may change to | it changes no type |
| `transitions` | Workflow transition **IDs** it may move through | it makes no status moves |

**The values must EXACTLY match your Jira** (case-sensitive). Read them from a
real ticket first:
```bash
BASE=<jira-base>; TOKEN=<bot-pat>
curl -sS "$BASE/rest/api/2/issue/<KEY>?fields=priority,issuetype" \
  -H "Authorization: Bearer $TOKEN" | jq '{priority:.fields.priority.name, type:.fields.issuetype.name}'
curl -sS "$BASE/rest/api/2/issue/<KEY>/transitions" \
  -H "Authorization: Bearer $TOKEN" | jq '.transitions[] | {id,name}'
```
**Apply:** `kubectl apply -f agent/deploy/k8s/config.yaml` (fast path — no rebuild).

### B) `dc/receiver.yaml` — behavior knobs (env vars)

These are plain environment variables on the receiver Deployment. Edit, then
`kubectl apply -f agent/deploy/k8s/dc/receiver.yaml && kubectl -n agents rollout restart deploy/agent-receiver`.

| Env | What it controls | Notes |
|---|---|---|
| `TRIGGER` | Which webhook flavor to expect | `jira-dc` for Data Center. Don't change. |
| `WEBHOOK_PATH` | The URL path Jira posts to | `/jira-webhook`. If you change it, update the Jira webhook URL to match. |
| `image:` / `AGENT_IMAGE` | The container image | **Both** must point at the same pushed tag (see 3.4). |
| `RUN_ENV` | Env passed to each run Job, comma-separated | The important one — see below. |
| `AUTHORIZED_ACTORS` | DC usernames allowed to trigger | Comma-separated. A label-add from anyone else is dropped. |

**`RUN_ENV` decoded** (this one trips people up — it's a single comma-joined string):
```
HARNESS=pi,MODEL=eu.anthropic.claude-sonnet-4-6,AWS_REGION=eu-west-1,GITLAB_BASE_URL=https://gitlab.brisa.internal
```
- `HARNESS=pi` — which coding agent CLI runs. Must match the image you built.
- `MODEL=eu.anthropic.claude-sonnet-4-6` — the Bedrock model. **Must match** the
  `bedrock_model_id` your terraform IAM policy allows, or Bedrock returns
  `AccessDenied`. Change all of them together if you switch models.
- `AWS_REGION=eu-west-1` — must be a region your Bedrock access covers.
- `GITLAB_BASE_URL=...` — the external GitLab URL the agent reads code from.
- To switch the agent to **basic auth** (if Jira PATs are disabled), append
  `,JIRA_AUTH_SCHEME=basic` here.

### C) `secrets.yaml` — the credentials

Kubernetes Secret, loaded into each run Job as env vars. Keys are
**UPPER_SNAKE_CASE on purpose** — a dash-cased key gets silently dropped and the
run fails with "JIRA_* is required".

| Key | What | For Brisa |
|---|---|---|
| `JIRA_API_TOKEN` | The bot's Jira credential | the DC **Personal Access Token** |
| `JIRA_EMAIL` | Bot username | only used if `JIRA_AUTH_SCHEME=basic` |
| `GITLAB_READ_TOKEN` | Read-only GitLab token | the deploy token |
| `WEBHOOK_HMAC_SECRET` | Verifies the webhook is really from Jira | `openssl rand -hex 32`; use the SAME value in the Jira webhook's "secret" field |

**Apply:** `kubectl apply -f agent/deploy/k8s/secrets.yaml && kubectl -n agents rollout restart deploy/agent-receiver`.

### D) `resourcequota.yaml` — how many runs at once

`pods: "6"` = `2 receiver replicas + up to 4 concurrent runs`. Raise the number
to allow more simultaneous triages. (This is a storm/cost guard, not a daily
budget — for a dollar ceiling use an AWS Budget.)

## Part 5 — A safe practice loop before you touch the customer's cluster

You don't need a cluster to test prompt/script logic. Two fast local loops:

**1. The script tests (proves your allowed-value edits + script shapes):**
```bash
bash agent/agents/jira-triage-dc/tests/run.sh      # expect PASS=20 FAIL=0
```
If you added a label/priority to the test config and a test, this catches typos.

**2. The runtime tests (proves the receiver/trigger logic):**
```bash
( cd agent/runtime && node --test )                # expect 54 pass / 0 fail
```

For the **prompt/rubric** itself, there's no local model call — you verify that on
a **throwaway Jira ticket** in the test project (Part 3.5). That's why the
deployment runbook insists on a test project before widening to real projects.

## Part 6 — Quick reference: "I want to change X"

| I want to… | Edit | Ship with | Rebuild image? |
|---|---|---|---|
| Change how the agent classifies / its instructions | `agents/jira-triage-dc/SKILL.md` (rubric body) | rebuild + push + new tag + apply receiver | **Yes** |
| Change the one-line task prompt | `agents/jira-triage-dc/SKILL.md` (`prompt:`) | same as above | **Yes** |
| Add/remove an allowed label/priority/type/transition | `deploy/k8s/config.yaml` | `kubectl apply -f config.yaml` | No |
| Add/remove an assignable person | `deploy/k8s/config.yaml` (`assignees`) | `kubectl apply -f config.yaml` | No |
| Change who can trigger | `deploy/k8s/dc/receiver.yaml` (`AUTHORIZED_ACTORS`) | apply + rollout restart | No |
| Change the model / region / GitLab URL | `deploy/k8s/dc/receiver.yaml` (`RUN_ENV`) | apply + rollout restart | No |
| Rotate a token | `deploy/k8s/secrets.yaml` | apply + rollout restart | No |
| Change a bundled script (jira.sh/gitlab.sh) | `agents/jira-triage-dc/scripts/*.sh` | rebuild + push + new tag + apply | **Yes** |
| Allow more concurrent runs | `deploy/k8s/resourcequota.yaml` | `kubectl apply` | No |

**The single most useful habit:** when a change "doesn't take effect," ask first
*"did this need an image rebuild, and did I bump the tag?"* (Part 3.4). That one
question explains 90% of the confusion when starting out.
