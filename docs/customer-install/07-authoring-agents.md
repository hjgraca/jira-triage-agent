# 07 — Authoring Agents (the skill IS the agent)

The listener is a **generic runner**: it doesn't know about "triage". What the
agent *does* is defined by the **skill** — specifically the YAML frontmatter of
its `SKILL.md`. To build a new agent, you write a skill; you don't touch the
listener code.

## Anatomy of a skill / agent

```
skills/<your-agent>/
  SKILL.md          frontmatter (the agent definition) + rubric body
  scripts/          bash tools the agent calls (optional)
```

`SKILL.md` frontmatter fields (read by
[agent-def.js](../../agent/runtime/listener/agent-def.js)):

```yaml
---
name: my-agent                 # identifier for logs
prompt: |                      # REQUIRED. The prompt sent to the harness.
  Do <the task> for {{key}}.   #   {{var}} placeholders are filled from the
  Act on this one item, stop.  #   trigger's parsed vars.
loopMarker: GENERATED BY AI    # sentinel the agent writes; loop guard drops echoes
authorizedActors: [acct-1]     # who may trigger (label-add path); ∪ AUTHORIZED_ACTORS env
trustTools: read,execute_bash  # harness tool grant (kiro-cli)
model: anthropic/claude-x      # optional model override
---
# My Agent
<rubric prose — how to do the task, the trust boundary, the output format>
```

Everything below the frontmatter is the **rubric body**: pi loads it via
`--skill`; skill-less harnesses (kiro-cli, opencode) get it inlined into the
prompt automatically.

## How the pieces connect

```
trigger parses the webhook ──▶ vars (e.g. { key: "KAN-5" })
                                  │
prompt template "...{{key}}..." ──┘ rendered ──▶ harness runs it
```

So a minimal new agent is: a new skill dir + set `AGENT_PATH` to it. Same
listener, same guardrails, different agent.

## Worked example: a PR-review agent on the generic trigger

1. **Write the skill** `skills/pr-review/SKILL.md`:
   ```yaml
   ---
   name: pr-review
   prompt: |
     Review merge request {{mr}} in repo {{repo}}. Use scripts/gitlab.sh to read
     the diff and post review comments. Be concise; stop when done.
   loopMarker: GENERATED BY AI REVIEW
   trustTools: read,execute_bash
   ---
   # PR Review
   <your review rubric...>
   ```
2. **Bake it into the image** (it's just another dir under `skills/`), or mount it.
3. **Configure the listener:**
   ```yaml
   - name: AGENT_PATH
     value: "/skills/pr-review"
   - name: TRIGGER
     value: "generic"        # signed POST whose body carries {mr, repo}
   ```
4. **Trigger it** with a signed POST:
   ```bash
   curl -X POST "$WEBHOOK_URL" \
     -H 'X-Triage-Token: <shared secret>' \
     -H 'X-Triage-Delivery-Id: pr-42' \
     -d '{"vars": {"repo": "acme/backend", "mr": "42"}}'
   ```

The generic trigger passes `vars` straight to the prompt template — no
Jira-specific parsing. For a non-trivial event source (GitHub webhooks with
their own signature + payload shape), write a trigger adapter instead — see
[trigger adapters](../../agent/runtime/trigger/index.js) and the harness
README's "add your own" pattern (triggers follow the same shape).

## What stays fixed no matter the agent

The runner's guardrails apply to every agent: request auth, dedupe, the spend
limiter (`MAX_CONCURRENT`/`SPAWN_CEILING`/`DAILY_BUDGET`), the watchdog, and the
egress NetworkPolicy. The loop guard and actor-allowlist apply on triggers that
support them (the Jira trigger does; the generic trigger trusts an authenticated
caller — see [Security](06-security.md)).

## Caveats

- **`prompt` is required.** A skill with no `prompt` in frontmatter fails fast at
  startup (a generic runner with no prompt is a misconfiguration).
- **The frontmatter parser is intentionally minimal** (scalars, `|` block
  scalars, simple lists) — enough for an agent definition, not general YAML.
- **One agent per listener instance.** `AGENT_PATH` selects one skill. To run
  several agents, run several Deployments (each its own skill + trigger), or
  front them with distinct webhook paths.
