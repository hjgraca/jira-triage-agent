# 05 — Operations

Running the agent day to day: verifying, monitoring, rotating credentials, tuning
cost, and troubleshooting. The runtime is **receiver → one Job per event**, so
most observability is plain `kubectl get jobs` / per-Job logs.

## Verify

Add the `triage` label to a low/medium test ticket. Expect:

1. The receiver logs an accept + Job create:
   ```json
   {"msg":"spawn","label":"KAN-5","trigger":"jira","agent":"jira-triage"}
   ```
2. A run Job appears and a pod runs:
   ```bash
   kubectl -n agents get jobs
   kubectl -n agents logs -l app.kubernetes.io/name=agent-run --tail=50
   ```
3. The agent posts a comment starting with
   `> *This was generated by AI during triage.*`, sets fields within your allowed
   sets, and **removes** the `triage` label (the label doubles as a work-queue flag).

A **high-severity** ticket instead gets a `needs-human` label + a recommendation
comment, with **no** field writes (the severity gate, R2c).

## Reading the logs

**Receiver** (one line per inbound event, structured JSON, no ticket bodies/PII):

| `msg` | Meaning |
|---|---|
| `listening` | receiver started |
| `spawn` | accepted → a run Job was created |
| `duplicate` | the Job already existed (409) — a redelivery; no second run |
| `drop` | gated out — `reason` (ineligible event, unauthorized label actor, loop-guard) |
| `reject` | `401`/`400` — `unauthenticated`, `bad-json` |
| `error` | `job-create-failed` — the API rejected the Job (check RBAC/quota) |

**Run** (each Job's own pod logs): the harness's own output, plus a non-zero exit
if it failed. `kubectl -n agents get jobs` shows COMPLETIONS; a failed Job retries
up to `backoffLimit` then stays Failed for inspection (until its TTL).

## Monitoring

Two label selectors do almost everything. They are easy to get wrong — the
receiver and run pods use `app.kubernetes.io/name`, **not** `app`:

| What | Selector |
|---|---|
| Receiver (ingress) pods | `app.kubernetes.io/name=agent-receiver` |
| Run (per-event) pods | `app.kubernetes.io/name=agent-run` |

**Is the agent alive and accepting events?** Tail the receiver and filter to the
decision lines — one per inbound webhook:

```bash
# live, both replicas, only the decision lines (spawn/drop/reject/duplicate/error)
kubectl -n agents logs -l app.kubernetes.io/name=agent-receiver -f --prefix \
  | grep --line-buffered -E '"msg":"(spawn|drop|reject|duplicate|error)"'

# last hour's activity at a glance
kubectl -n agents logs -l app.kubernetes.io/name=agent-receiver --since=1h --tail=500 \
  | grep -E '"msg":"(spawn|drop|reject|duplicate|error)"'
```

A **silent** receiver (no lines at all when you add a label) means the trigger
never reached it — go to the rule's audit log, not the cluster. See
[Troubleshooting](#troubleshooting).

**Are runs succeeding?** Jobs are the run ledger; finished pods linger (until TTL)
so you can read them:

```bash
kubectl -n agents get jobs --sort-by=.metadata.creationTimestamp   # COMPLETIONS 1/1 = ok
kubectl -n agents get pods -l app.kubernetes.io/name=agent-run     # Completed / Running / Error

# follow whatever is running right now
kubectl -n agents logs -l app.kubernetes.io/name=agent-run -f --tail=50 --prefix

# triage verdict of one run (the agent's closing summary)
kubectl -n agents logs job/<job-name> --tail=200 | grep -oE '"text":"[^"]*(complete|category|Summary)[^"]*"' | tail -3
```

**One-shot health pulse** (receiver up + recent run outcomes):

```bash
kubectl -n agents get deploy agent-receiver                        # READY n/n
kubectl -n agents get jobs --sort-by=.metadata.creationTimestamp | tail -10
```

**Ground truth is in Jira, not the cluster.** A run "Complete" only means the
harness exited 0 — confirm it actually triaged by checking the ticket got the
`> *This was generated by AI during triage.*` comment and the `triage` label was
removed. The fastest full-chain check (CloudFront → receiver → Job → Jira comment)
is the live e2e test, which creates a throwaway ticket and asserts each link:

```bash
bash agent/deploy/test/e2e.sh          # or: --step to walk through each stage
```

For continuous/at-a-glance monitoring, point a dashboard at the run Jobs' success
ratio and the receiver's `reject`/`error` line rate (both are in the pod logs that
ship to CloudWatch via the cluster's log driver).

## Credential rotation

Rotate the Jira token, GitLab token, and the webhook/automation secret on a
schedule (≤90 days):

1. Revoke the old credential at the source.
2. Generate the new one.
3. Update `agent/deploy/k8s/base/secrets.yaml` (and, for the **HMAC** path, the Jira
   system-webhook secret to match).
4. Apply, and roll the receiver (run Jobs read the Secret fresh each time, so they
   need no roll):
   ```bash
   kubectl apply -f agent/deploy/k8s/base/secrets.yaml
   kubectl -n agents rollout restart deploy/agent-receiver
   ```

If a token ever appears in logs, treat it as compromised and rotate immediately.

## Cost control

Each accepted event creates one billable run. The bounds are now **Kubernetes**,
not in-process:

| Bound | Where | Caps |
|---|---|---|
| `pods` | `resourcequota.yaml` | concurrent runs (storm/loop defense). Counts non-terminal pods only, so finished runs free their slot immediately; the cap is `receiver replicas + max concurrent runs` (it's namespace-wide). |
| `activeDeadlineSeconds` | the Job spec (`lib/job.js`) | wall-clock per run |
| `backoffLimit` | the Job spec | retries on failure |
| **AWS Budgets / Bedrock quota** | provider-side | cumulative dollar spend (the daily backstop) |

There is no in-app rolling-24h run counter anymore — it needed shared state. Use
an AWS Budget for the hard dollar ceiling. See [Security](06-security.md).

## Scaling

The receiver is **stateless** — scale it freely (`replicas` in `receiver.yaml`).
Dedupe and concurrency live in Kubernetes (Job name + ResourceQuota), not in any
pod, so there's no single-writer constraint.

## Troubleshooting

| Symptom | Likely cause | Check |
|---|---|---|
| Adding label does nothing; **no** receiver log | Trigger didn't send | Cloud: Automation rule audit log; DC: webhook delivery history. Confirm condition `Labels contains triage` + the URL. |
| Audit log: *Send web request* → `500` with a **Squid** "could not be retrieved" page | **Stale webhook URL** — CloudFront was recreated and got a new domain, but the rule still points at the old one (which no longer resolves) | `terraform -chdir=workshop/terraform output -raw triage_webhook_url` and set the rule's URL to match. The 500/Squid page comes from Jira's egress proxy failing to reach a dead host, **not** from the receiver. |
| `reject reason:"unauthenticated"` | Wrong/missing auth | Cloud: `X-Triage-Token` matches `AUTOMATION_SHARED_SECRET`. DC: HMAC secret matches; `sha256=` prefix. |
| `error reason:"job-create-failed"` | RBAC or quota | `rbac.yaml` applied + receiver uses `agent-receiver` SA; or the `pods` quota is full. |
| `drop reason:"unauthorized label actor"` | Initiator not allowlisted | Add their accountId to `AUTHORIZED_ACTORS`, re-apply, roll the receiver. |
| `drop reason:"ineligible event"` | DC update without a label-add in the changelog | Expected — only label-adds (and creates) are eligible. |
| Job created but pod **Pending** | ResourceQuota full | Runs are queued; raise `pods` (remember it includes the receiver replicas) or wait. |
| Run Job **Failed** | harness non-zero exit | `kubectl -n agents logs job/<name>`; common causes are bad creds or an out-of-set write. |
| Agent comments but **doesn't assign** | No mapped owner / empty `assignees` | Expected when no CODEOWNERS owner maps to an allowed assignee — it recommends instead. |
| `exec format error` | Wrong image arch | Rebuild `--platform linux/amd64`. |

## Pause / disable

- **Soft pause:** disable the Jira Automation rule (or system webhook) — nothing
  in the cluster changes.
- **Hard stop:** `kubectl -n agents scale deploy/agent-receiver --replicas=0`
  (no new Jobs are created; in-flight Jobs finish).
