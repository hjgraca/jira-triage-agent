# 05 — Operations

Running the agent day to day: verifying, monitoring, rotating credentials, tuning

## Verify

Add the `triage` label to a low/medium test ticket (or create one, if you wired
the create-time trigger). Expect:

1. The listener logs a spawn line:
   ```json
   {"msg":"spawn","key":"KAN-5","authVia":"shared-secret"}   // or "hmac"
   ```
2. The agent posts a comment starting with
   `> *This was generated by AI during triage.*`
3. Fields are set within your allowed sets, and the `triage` label is **removed**
   (the label doubles as a work-queue flag).

A **high-severity** ticket instead gets a `needs-human` label + a recommendation
comment, with **no** field writes (the severity gate, R2c).

Watch logs live:

```bash
POD=$(kubectl -n triage get pod -l app.kubernetes.io/name=triage-listener -o jsonpath='{.items[0].metadata.name}')
kubectl -n triage logs -f "$POD"
```

## Reading the logs

The listener emits structured JSON, **no ticket bodies or PII**. Key events:

| `msg` | Meaning |
|---|---|
| `listening` / `ready` | startup; `ready` includes the resolved bot accountId |
| `spawn` | accepted → a `pi` run started (`authVia` = `hmac` or `shared-secret`) |
| `triage_done` / `triage_exit` | the run finished (`code`, `toolError`) |
| `drop` | gated out — `reason` tells you which guard (duplicate, ineligible event, unauthorized label actor, loop-guard, limiter:*) |
| `reject` | `401`/`400` — `unauthenticated`, `bad-json`, etc. |

> **`toolError:true` on a `code:0` exit is usually a guardrail firing, not a
> bug** — e.g. the agent tried to remove a `triage` label that wasn't present, or
> an allowed-value gate rejected an out-of-set write. Confirm against the actual
> ticket before treating it as a failure.

## Credential rotation

Rotate the Jira token, GitLab token, and the webhook/automation secret on a
schedule (≤90 days). For each:

1. Revoke the old credential at the source (Jira/GitLab).
2. Generate the new one.
3. Update `agent/deploy/k8s/triage-secrets.yaml` (and, for the **HMAC** path, the Jira
   system-webhook secret to match).
4. Apply and roll:
   ```bash
   kubectl apply -f agent/deploy/k8s/triage-secrets.yaml
   kubectl -n triage rollout restart deploy/triage-listener
   ```

If a token ever appears in logs, treat it as compromised and rotate immediately.

## Cost control

Each accepted webhook spawns a billable Bedrock run. Three bounds, all on the
listener (env in `agent/deploy/k8s/triage-listener.yaml`):

| Knob | Default | Bounds |
|---|---|---|
| `MAX_CONCURRENT` | 3 | simultaneous runs |
| `SPAWN_CEILING` | 60 | runs per rolling minute (storm/loop defense) |
| `DAILY_BUDGET` | 500 | runs per rolling 24h (hard cost cap) |

When a bound is hit the webhook is dropped with a logged `200` (`drop reason:
limiter:*`) — Jira isn't told to retry forever.

## Scaling note

The Deployment is **pinned to one replica** with a `Recreate` strategy. Dedupe
and the spend limiter are in-memory and per-pod — two replicas would each dedupe
and rate-limit independently, letting duplicates and over-budget spend slip
through. Scaling out requires moving that state to a shared store first; don't
bump `replicas` without it.

## Troubleshooting

| Symptom | Likely cause | Check |
|---|---|---|
| Pod never **Ready** | `/myself` failing | `JIRA_BASE_URL`/`JIRA_EMAIL`/`JIRA_API_TOKEN`; egress to Jira; `kubectl logs` for `startup_myself_failed` |
| Adding label does nothing; **no** log line | Trigger didn't send | Cloud: the Automation rule's audit log; DC: webhook delivery history. Confirm condition `Labels contains triage` and the URL. |
| `reject reason:"unauthenticated"` | Wrong/missing auth | Cloud: `X-Triage-Token` matches `automation-shared-secret`. DC: HMAC secret matches; `sha256=` prefix. |
| `drop reason:"unauthorized label actor"` | Initiator not allowlisted | Add their accountId to `AUTHORIZED_ACTORS`, re-apply, roll. |
| `drop reason:"ineligible event"` | DC update without a label-add in the changelog | Expected — only label-adds (and creates) are eligible. |
| `drop reason:"limiter:daily-budget"` | Daily cap reached | Raise `DAILY_BUDGET` or wait for the 24h window to roll. |
| Agent comments but **doesn't assign** | No mapped owner / empty `assignees` | Expected when no CODEOWNERS owner maps to an allowed assignee — it recommends instead. |
| `crashloop` / `exec format error` | Wrong image arch | Rebuild `--platform linux/amd64`. |

## Pause / disable

- **Soft pause:** disable the Jira Automation rule (or the system webhook). The
  agent stops receiving triggers; nothing in the cluster changes.
- **Hard stop:** `kubectl -n triage scale deploy/triage-listener --replicas=0`.
